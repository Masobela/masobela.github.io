Projects & What I Did

PROJECT — 001
Static Website on AWS S3
Deployed a fully hosted static website using core AWS services — no server required.
Step-by-step
01Created an S3 bucket and enabled static website hosting in the bucket settings+
S3 (Simple Storage Service) lets you store files in the cloud. By enabling "Static Website Hosting" in the bucket properties, AWS serves your HTML files directly as a website — no server needed. You set the index.html as the entry point.
02Uploaded HTML, CSS, and JS files and set the index document+
I uploaded all the website files into the S3 bucket using the AWS Console. The index document tells AWS which file to load first when someone visits the URL — usually index.html. This is how your browser knows where to start.
03Wrote a bucket policy to allow public read access to all objects+
By default, S3 buckets are private. I wrote a JSON bucket policy using IAM permissions that grants s3:GetObject access to everyone (*). This makes the files publicly readable so visitors can load the website in their browsers.
04Created a CloudFront distribution pointing to the S3 bucket for fast global delivery+
CloudFront is AWS's CDN (Content Delivery Network). It caches your website files at edge locations around the world, so users in Johannesburg, London, or Tokyo all get fast load times. I created a distribution with the S3 bucket as the origin.
05Configured Route 53 to point a domain name to the CloudFront distribution+
Route 53 is AWS's DNS service. I created an A record (alias) that maps a custom domain name to the CloudFront distribution URL. This means visitors can type a friendly domain instead of a long AWS URL to reach the site.

PROJECT — 002
HPE RESTful Web Service
Built a functioning REST API with data upload and unit test coverage during the HPE Forage simulation.
Step-by-step
01Reviewed the brief and designed the API structure+
I read the HPE project brief carefully to understand what the web service needed to do. I then planned the API endpoints, deciding which HTTP methods (GET, POST) to use and what data each endpoint would accept and return.
02Built the RESTful web service with GET and POST endpoints+
REST (Representational State Transfer) is an architectural style for APIs. I implemented GET endpoints to retrieve data and POST endpoints to receive and store new data. Each endpoint returns appropriate HTTP status codes (200 OK, 201 Created, 400 Bad Request).
03Implemented data upload capability through the API+
I added the ability for clients to upload data via POST requests. The service receives the data in the request body, validates it, and stores it. I learned about request/response cycles, JSON payloads, and how APIs handle data persistence.
04Wrote unit tests for every endpoint+
Unit tests verify that each piece of code works correctly in isolation. I wrote tests that sent requests to each endpoint with different inputs (valid data, invalid data, edge cases) and asserted the responses were correct. This ensures the service is reliable and safe to deploy.
05Reviewed and submitted matching HPE engineering standards+
Before submitting I reviewed the code for clarity, structure, and alignment with the brief. HPE's simulation is designed to mirror real engineering expectations — clean code, proper error handling, and well-documented logic that a team could maintain.

PROJECT — 003
Wells Fargo Data Model
Designed and implemented a data model for a financial application in the Wells Fargo engineering simulation.
Step-by-step
01Analysed business requirements for the data model+
I read the Wells Fargo brief to understand what financial data needed to be captured. In banking, data modelling is critical — the wrong structure can cause serious issues. I identified the entities (things to track), their attributes (properties), and how they relate to each other.
02Designed entities, relationships, and attributes+
A data model defines the structure of your data. I mapped out entities (like customers, accounts, transactions), their attributes (name, balance, date), and relationships between them (a customer has many accounts). This is the foundation for any database or application.
03Implemented the model in code and validated with sample data+
I translated the design into code — creating classes/structures that represent each entity and their relationships. I then ran sample data through the model to confirm it captured and stored information correctly without errors or data loss.
04Tested data integrity across multiple scenarios+
Data integrity means the data stays accurate and consistent. I tested edge cases — what happens with missing fields, duplicate entries, or invalid values? In financial applications this is non-negotiable, so I made sure the model handled every scenario correctly.
